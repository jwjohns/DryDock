Drydock

Separating the "Hull" (Static Infrastructure) from the "Cargo" (Dynamic Configuration) across GCP and Azure.

Mission

What is Drydock?

Drydock is an open source deployment framework designed to separate static infrastructure ("the hull") from dynamic configuration and secrets ("the cargo"), enabling secure, scalable, and reproducible deployments across GCP and Azure (initially) using GitHub Actions, Terraform, Helm, and Workload Identity Federation (WIF).

It's purpose-built for modern teams moving away from legacy Jenkins-based deploys, toward zero-key, auditable, Git-native automation.

Core Philosophy

Hull vs. Cargo

In a shipyard drydock, the hull is constructed before it's ever launched into water. Drydock adopts this metaphor:

  • The Hull = statically defined, versioned infrastructure & Helm templates (managed in Git).
  • The Cargo = runtime values, secrets, and overrides injected securely at deployment time (fetched from Secret Manager, Key Vault, GCS, Blob Storage, etc.).

This separation ensures your infrastructure code is clean and secrets are never stored in your repository.

Key Features

Why Use Drydock?

  • Workload Identity Federation (WIF): Eliminate long-lived cloud credentials in CI/CD for GCP and Azure.
  • Environment-specific Layering: Use Terraform .tfvars and Helm overrides for distinct configurations per environment.
  • Secure Runtime Injection: "Cargo" (secrets, runtime config) is fetched and injected only at deploy-time.
  • GitHub Actions Native: Designed with composable GitHub Actions workflows at its core.
  • Multi-Cloud Ready: Supports parallel definitions for GCP and Azure infrastructure.
  • Fleet Deployment Support: Structure adaptable for deploying across many stores, regions, or environments.

How it Works

Simplified Workflow

  1. Commit static templates (Terraform Hulls, Helm Charts) to Git.
  2. Trigger GitHub Actions (on push, tag, or manual dispatch).
  3. Action authenticates to the target cloud (GCP/Azure) using WIF.
  4. Action fetches runtime "Cargo" (.tfvars, Helm overrides, secrets) from secure sources (Secret Manager, Key Vault, GCS, etc.).
  5. Action runs terraform apply using the appropriate Hull + Cargo.
  6. Action runs helm upgrade --install using the base chart + Hull values + Cargo overrides.
